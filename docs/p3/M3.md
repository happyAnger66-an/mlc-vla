# M3：端到端闭环 + PyTorch bit-exact 对拍 + 输出反解

M3 在 M0（GPT 主干 + KV-cache `prefill/decode_step`）、M1（视觉 tokenizer `embed_visual` + 宿主 AR 环）、
M2（图内 `decode_loop_kv` + cuBLAS/CUDA Graph + 主干/头 loader）之上，打通 **端到端离散输出闭环**
并给出 **bit-exact 数值对拍**，收口 TVM 侧可交付范围。

> 总体方案见 `docs/p3/arch.md`；M0/M1/M2 见同目录。照搬 mlc-vla（π0.5）姿势收尾。

## 0. 关键澄清：TVM engine 边界（为什么 encode 外层不进 TVM）

复核参考实现后确认两点，据此界定「剩余部分」：

1. **`TrajTokenizer.ids_to_embed` == `embed_tokens(ids)` 查表**（`trajectory_encoder_v2.py:289`）。
   即 M0 的 `embed` 占位**结构本就正确**——不是替身，只差把真实权重接进来。
2. **encode 外层编排按 arch.md 设计属宿主/PVA 预处理，不是 TVM 固定-shape 图**：
   多相机 DINOv3 外层拼接、view_token/PE、时序 `hist_img_feat` 缓存（`visual_encoder.py`）、
   navi 车道位标解码（`navigation_encoder_2.py` 的 `torch.remainder/div`）、
   history 码本最近邻 `cdist.argmin`（`trajectory_encoder.py`）、数据相关 crop/resize——
   这些是数据相关、非定尺寸的宿主逻辑。**TVM engine 只吃 DINOv3 backbone（`embed_visual`，M1 已交付）+ 主干 prefill/decode**。

因此 TVM 路径的真正剩余项是：真实 traj 嵌入权重、离散输出反解、端到端编排、数值对拍——本 milestone 全部补齐。

## 2. 输出反解 `plannn3_decode.py`

离散 token 是 TVM 的主输出（对齐 `infer.py` 的 golden）；反解成连续 waypoints 是数据无关的宿主后处理：

```text
traj_ids [1, pred_times]
  └─ split: meta_action_ids (前 meta_action_size 个) | main_action_ids (末 main_action_length=15 个)
  └─ PCATrajDecoder.decode(main): 反量化 → Z@V+mu → 反 scale/归一 → 增量 [B,T,3] (dx,dy,dyaw)
  └─ reconstruct_waypoints:  yaw=cumsum(dyaw); line_yaw=[Δ0, Δ1:+yaw:-1];
                             x,y=cumsum(len·cos/sin(line_yaw)) → 绝对 (x,y,yaw)
```

- `PCATrajDecoder` 从与训练一致的 `*_pca_tokenizer.json`（+ 同目录 basis npy）加载；无 json 时 `decode_traj_ids(pca=None)` 仅做 meta/main 拆分。
- `reconstruct_waypoints` 与 `TrajTokenizer.decode` 第二段完全对齐（见 §4 验证）。

## 3. 端到端编排 `Plannn3Runner.run`

```python
runner = Plannn3Runner(cfg, "cuda", cuda_graph=True)   # 或 target="c" 冒烟
runner.set_params(...)                                  # loader 加载真实权重
out = runner.run(token_embeds, use_graph=True, pca=pca) # token_embeds 来自 encode 阶段
# out = {traj_ids, meta_action_ids, main_action_ids, [wp_delta, waypoints]}
```

`token_embeds` 的来源：DINOv3 backbone 走 TVM `embed_visual`（M1），多视角/时序/navi/history 外层编排在宿主侧拼接（§0）。

## 4. 验证（CPU `target=c`，无需 GPU/NFS/权重）

```bash
export TVM_FFI_DISABLE_TORCH_C_DLPACK=1     # 关键：绕过 torch C-DLPack JIT，import ~1.7s
python -m mlc_vla.compare_p3 --dummy --target c --graph
```

实测结果：

```
[compare] host-step: ref=[23, 9, 3, 20]
[compare] host-step: tvm=[23, 9, 3, 20]
[compare] traj_ids bit-exact (ref vs TVM host-step): True
[compare] graph : tvm=[23, 9, 3, 20]
[compare] traj_ids bit-exact (ref vs TVM decode_loop_kv): True
PASS
```

- **traj_ids bit-exact**：PyTorch 参考主干（`network.py` GPT 的自包含复刻：interleaved RoPE / LayerNorm-no-bias /
  GELU-erf / causal，解码走 `Net.decode` 式每步重算全序列）与 TVM 逐步环、TVM 图内 `decode_loop_kv` 三者完全一致。
- **waypoint 重建**：`reconstruct_waypoints` vs torch 端口 max_abs_diff = `2.2e-16`（机器精度）。
- **端到端 `run`**：宿主环与图内环 `traj_ids` 一致；`meta/main` 按 `18=3+15` 正确拆分。
- **loader**：`embed.weight` 现映射到 `traj_encoder.embed_tokens.weight`，对拍中随参数一并加载生效。

> 对拍参考选用自包含 torch 主干（而非完整 NIO `Net`），因后者依赖 NFS 权重 + 全套编码器 + timm；
> 待 GPU + 真实 checkpoint 就位时，可用同一 `token_embeds` 接入 `infer.py` 的 `Net.decode` 做整网 golden 复核。
